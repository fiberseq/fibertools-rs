# release-please Migration Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Move `fibertools-rs` + `molecular-annotation` releases to release-please monorepo mode, with cargo-dist attaching binaries to release-please's GitHub Release and `publish-crates.yml` publishing to crates.io.

**Architecture:** release-please (manifest mode, two components, path-based routing) owns versions, changelogs, tags, and GitHub Releases. cargo-dist switches to `create-release = false` and only uploads fibertools binaries to the existing Release. The already-present `publish-crates.yml` publishes both crates to crates.io on release publish. git-cliff and manual `cargo release`/tagging are retired.

**Tech Stack:** release-please-action@v4 (`release-type: rust`, `cargo-workspace` plugin), cargo-dist 0.30.3, GitHub Actions, crates.io Trusted Publishing.

**Spec:** `docs/superpowers/specs/2026-07-15-release-please-migration-design.md`

## Global Constraints

- cargo-dist version is pinned at **0.30.3** (matches `dist-workspace.toml` `cargo-dist-version`). Use exactly this version for `dist generate` to avoid spurious workflow diffs.
- fibertools-rs release tag stays **`v0.X.Y`** (root component sets `include-component-in-tag: false`). Do not change the tag shape.
- `molecular-annotation` is a **library with no binaries** → `dist = false`.
- Current versions: `fibertools-rs = 0.10.0`, `molecular-annotation = 0.0.1`. The dependency pin is `[workspace.dependencies] molecular-annotation = { path = "molecular-annotation", version = "0.0.1" }` (`^0.0.1` == exact `0.0.1`).
- Commits follow Conventional Commits.
- Must NOT break: the GitHub Release, attached binaries, the shell installer, or the crates.io publish.
- All work happens on branch `ci/crates-io-publish` (already checked out; contains the crates.io publish workflow + lockstep-decoupling changes + the spec).

### Task 1: release-please config, manifest, and workflow

**Files:**
- Create: `release-please-config.json`
- Create: `.release-please-manifest.json`
- Create: `.github/workflows/release-please.yml`

**Interfaces:**
- Produces: two release-please components — `fibertools-rs` (path `.`, tag `v<version>`) and `molecular-annotation` (path `molecular-annotation`, tag `molecular-annotation-v<version>`). Consumed by Task 2 (cargo-dist triggers on the `v<version>` tag) and Task 3 (`publish-crates.yml` triggers on the published Release).

- [ ] **Step 1: Create `release-please-config.json`**

```json
{
  "$schema": "https://raw.githubusercontent.com/googleapis/release-please/main/schemas/config.json",
  "plugins": ["cargo-workspace"],
  "packages": {
    ".": {
      "release-type": "rust",
      "component": "fibertools-rs",
      "include-component-in-tag": false,
      "exclude-paths": ["molecular-annotation"],
      "draft": true
    },
    "molecular-annotation": {
      "release-type": "rust",
      "component": "molecular-annotation"
    }
  }
}
```

- [ ] **Step 2: Create `.release-please-manifest.json`**

```json
{
  ".": "0.10.0",
  "molecular-annotation": "0.0.1"
}
```

- [ ] **Step 3: Create `.github/workflows/release-please.yml`**

```yaml
name: release-please

on:
  push:
    branches: [main]

permissions:
  contents: write
  pull-requests: write

jobs:
  release-please:
    runs-on: ubuntu-latest
    steps:
      - uses: googleapis/release-please-action@v4
        with:
          # Manifest mode: reads release-please-config.json +
          # .release-please-manifest.json from the repo root.
          config-file: release-please-config.json
          manifest-file: .release-please-manifest.json
```

- [ ] **Step 4: Validate the JSON and YAML parse**

Run:
```bash
python3 -c "import json; json.load(open('release-please-config.json')); json.load(open('.release-please-manifest.json')); print('json ok')"
python3 -c "import yaml,sys; yaml.safe_load(open('.github/workflows/release-please.yml')); print('yaml ok')"
```
Expected: prints `json ok` then `yaml ok` with no traceback.

- [ ] **Step 5: Commit**

```bash
git add release-please-config.json .release-please-manifest.json .github/workflows/release-please.yml
git commit -m "ci: add release-please manifest config and workflow"
```

- Consumes: the `v<version>` tag produced by Task 1.

### Task 3: Add a concurrency guard to `publish-crates.yml`

**Files:**
- Modify: `.github/workflows/publish-crates.yml`

**Interfaces:**
- Consumes: the published Release(s) from Task 1 (two Releases — fibertools + MA — can publish from one release-PR merge).

- [ ] **Step 1: Add a `concurrency` block after the `permissions` block**

Insert into `.github/workflows/publish-crates.yml` immediately after the `permissions:` block:
```yaml

- [ ] **Step 2: Replace the "Cutting a release" section of `CONTRIBUTING.md`**

Replace everything from the `## Cutting a release` heading to the end of the file with:
```markdown
## Cutting a release

Releases are automated with
[release-please](https://github.com/googleapis/release-please). You do not bump
versions or create tags by hand.

1. Merge PRs to `main` using [Conventional
   Commits](https://www.conventionalcommits.org/) (`feat:`, `fix:`, `perf:`,
   `feat!:`/`BREAKING CHANGE:` for majors). Squash-merge is recommended so each
   PR is one logical change.
2. release-please keeps a **release PR** open that bumps versions and updates
   the changelogs. Which crate is bumped is determined by the file paths a
   commit touches: changes under `molecular-annotation/` bump
   `molecular-annotation`; changes elsewhere bump `fibertools-rs`.
3. When you are ready to release, merge the release PR. That creates the git
   tag(s) and GitHub Release(s).
4. On the `v0.X.Y` tag, `cargo dist` builds the `ft` binaries and installer and
   uploads them to the release-please Release. `publish-crates.yml` then
   publishes the crates to crates.io via Trusted Publishing.

Changelogs (`CHANGELOG.md` for `fibertools-rs`,
`molecular-annotation/CHANGELOG.md` for the library) are generated by
release-please — do not edit them by hand.

### Re-triggering a failed release

If the `release.yml` (cargo-dist) run fails after the tag exists, fix the issue
(e.g. update `cargo-dist-version` in `dist-workspace.toml`, then run `dist
generate`), commit, and re-run the failed workflow from the Actions tab, or
delete the tag/Release and re-run release-please:

```bash
gh release delete v0.X.Y --yes
git push origin :refs/tags/v0.X.Y
```
```

### Task 5: Validate the flow end-to-end (GitHub-side checkpoint)

This task proves the five risk items from the spec. It runs against the GitHub
repo and is observational — no code is written. Do it on a scratch branch /
pre-release before relying on the flow. It requires the branch merged to `main`
(or pushed and the workflow run against it) and a crates.io token for the
one-time `molecular-annotation` bootstrap.

**Interfaces:** none (validation only).

- [ ] **Step 1: Dry-run release-please routing locally (optional but recommended)**

Run (needs Node and a GitHub token with repo read):
```bash
GITHUB_TOKEN=$(gh auth token) npx release-please@latest manifest-pr \
  --repo-url=fiberseq/fibertools-rs \
  --dry-run --debug 2>&1 | tail -40
```
Expected: the dry-run lists the components it would release and shows that a
change under `molecular-annotation/**` maps to the `molecular-annotation`
component and NOT to `fibertools-rs` (verifies `exclude-paths`). If MA-only
changes also show up under `fibertools-rs`, the `exclude-paths` config is wrong
— fix Task 1 before proceeding.

- [ ] **Step 2: Push the branch and open a PR**

```bash
git push -u origin ci/crates-io-publish
gh pr create --fill --base main
```
Expected: PR created. CI (CI.yml) runs; release-please.yml does not act until merged to main.

- [ ] **Step 3: Merge to main and confirm the release PR appears**

After merging the PR to `main`, check the Actions tab for the `release-please`
run and confirm it opens a release PR.
Expected: a release PR titled like `chore(main): release ...` appears, bumping
`fibertools-rs` (and `molecular-annotation` only if its files changed), with
CHANGELOG entries. Confirm the fibertools bump is proposed as tag `v0.X.Y`
(no `fibertools-rs-` prefix) — verifies `include-component-in-tag: false`.

- [ ] **Step 4: Bootstrap the first `molecular-annotation` publish (one-time)**

crates.io Trusted Publishing cannot publish a brand-new crate. Before the first
automated run can publish `molecular-annotation`, publish it once with a token:
```bash
cargo login          # paste a crates.io token (scopes: publish-new + publish-update)
cargo publish -p molecular-annotation --locked
```
Then, on crates.io, configure Trusted Publishing for BOTH `fibertools-rs` and
`molecular-annotation` (repo `fiberseq/fibertools-rs`, workflow
`publish-crates.yml`).
Expected: `molecular-annotation 0.0.1` appears on crates.io.

- [ ] **Step 5: Merge the release PR and observe the full chain**

Merge the release PR. Watch the Actions tab.
Expected, in order:
1. release-please creates tag `v0.X.Y` + a **draft** GitHub Release with notes
   (verifies `"draft": true` on the fibertools component). The MA release (if
   MA changed) is created **published**, not draft.
2. `release.yml` (cargo-dist) runs on the tag, builds `ft` binaries + installer,
   uploads them to that draft Release, then un-drafts it (does NOT create a
   second Release). Confirm the Release ends up published with binaries
   attached — verifies the draft handshake.
3. The `molecular-annotation-*` tag/release (if MA changed) is a cargo-dist
   no-op (no artifacts, no errored jobs) — verifies `dist = false`.
4. `publish-crates.yml` runs when the fibertools release is un-drafted (and
   immediately for MA) and publishes fibertools (and MA if changed) to
   crates.io; already-published versions are skipped.

- [ ] **Step 6: Confirm the release artifacts and crates**

```bash
gh release view v0.X.Y --json assets --jq '.assets[].name'
cargo search fibertools-rs | head -1
cargo search molecular-annotation | head -1
```
Expected: the Release lists the platform archives + installer; `cargo search`
shows the new `fibertools-rs` version and `molecular-annotation` on crates.io.
