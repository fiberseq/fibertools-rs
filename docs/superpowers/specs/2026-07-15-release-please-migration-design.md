# Release automation: release-please monorepo + cargo-dist + crates.io

**Date:** 2026-07-15
**Status:** Approved design, pending implementation plan

## Goal

Move `fibertools-rs` releases to a conventional-commit-driven flow where
[release-please](https://github.com/googleapis/release-please) owns versioning,
changelogs, tags, and GitHub Releases (per crate), while the existing tooling
keeps its current jobs:

- **cargo-dist** builds the `fibertools-rs` binaries + shell installer and
  attaches them to the GitHub Release.
- **`publish-crates.yml`** publishes both crates to crates.io via Trusted
  Publishing.

After the migration, a release happens by merging the release PR that
release-please keeps open — no manual version bumping or tagging.

## Context / current state

- Workspace: `fibertools-rs` (binary `ft`, at repo root, currently `0.10.0`)
  and `molecular-annotation` (library, `molecular-annotation/`, `0.0.1`).
  Versions are **decoupled**: fibertools depends on MA via
  `[workspace.dependencies] molecular-annotation = { path = ".../molecular-annotation", version = "0.0.1" }`.
  For a `0.0.x` crate a caret requirement is an exact pin (`^0.0.1` == `0.0.1`),
  so each fibertools release is welded to a specific MA release.
- Releases today: manual `cargo release {type}` or manual commit + `git tag
  vX.Y.Z`, then `git push --tags`. cargo-dist (`.github/workflows/release.yml`,
  autogenerated) triggers on the tag, builds artifacts, and **creates** the
  GitHub Release with a body generated from **git-cliff** (`cliff.toml`).
- The cargo-dist `announce` job posts to **no external service** (no
  Slack/Discord/Mastodon). "Announcement" == the GitHub Release itself. The
  only things that must not break: the GitHub Release, the attached binaries,
  and the shell installer.
- `publish-crates.yml` (already added on branch `ci/crates-io-publish`)
  publishes `molecular-annotation` then `fibertools-rs` on `release: published`
  or manual dispatch, using crates.io Trusted Publishing (OIDC). It is
  idempotent: an already-published version is treated as success.
- Commits already follow Conventional Commits (`feat!`, `fix(docs)`, `perf`,
  `chore`, ...).

## Chosen approach

**release-please owns the release; cargo-dist attaches binaries** (evaluated
against two alternatives — see "Alternatives considered").

Use release-please **manifest (monorepo) mode** with **two components** and
independent per-crate versions. Commit-to-package routing is by **diff file
paths**, not commit scopes.

### 1. release-please configuration (two new files)

`release-please-config.json`:

```json
{
  "$schema": "https://raw.githubusercontent.com/googleapis/release-please/main/schemas/config.json",
  "plugins": ["cargo-workspace"],
  "packages": {
    ".": {
      "release-type": "rust",
      "component": "fibertools-rs",
      "include-component-in-tag": false,
      "exclude-paths": ["molecular-annotation"]
    },
    "molecular-annotation": {
      "release-type": "rust",
      "component": "molecular-annotation"
    }
  }
}
```

`.release-please-manifest.json` (seeds current versions):

```json
{
  ".": "0.10.0",
  "molecular-annotation": "0.0.1"
}
```

Key points:

- **Path routing:** a commit touching only `molecular-annotation/**` bumps MA;
  a commit touching `src/**` (etc.) bumps fibertools; a commit touching both
  bumps both. `exclude-paths: ["molecular-annotation"]` on the root component
  (path `.`) is load-bearing — without it the root package matches *every* file
  and MA-only changes would also bump fibertools. `feat(ma)`/`feat(ft)` scopes
  become optional changelog flavor, not a required routing mechanism.
- **`cargo-workspace` plugin:** when MA bumps, this updates fibertools's
  `[workspace.dependencies]` requirement (and bumps dependents as needed) so the
  exact pin stays in sync automatically.
- **Two changelogs (accepted):** release-please maintains one CHANGELOG per
  component — root `CHANGELOG.md` (fibertools) and
  `molecular-annotation/CHANGELOG.md`. It has no clean single-file merge; the
  monorepo split is the accepted tradeoff for independent auto-versioning.
- **Tag format:** `include-component-in-tag: false` on the root component keeps
  the fibertools tag as **`v0.X.Y`** (unchanged from today), so cargo-dist's
  trigger and `publish-crates.yml` need no changes. MA tags as
  `molecular-annotation-v0.0.x`.

### 2. New workflow `.github/workflows/release-please.yml`

- Trigger: `push` to `main`.
- `googleapis/release-please-action@v4` in manifest mode (reads the two config
  files above).
- Permissions: `contents: write`, `pull-requests: write`.
- Behavior: maintains the release PR (version bumps + CHANGELOG updates).
  Merging that PR creates the tag(s) + GitHub Release(s).

### 3. Release flow (end-to-end)

1. Contributors merge Conventional-Commit PRs to `main` (squash recommended:
   one commit per PR == one clean logical change to route by path).
2. release-please keeps a release PR open with pending bumps + changelog.
3. Merging the release PR → release-please creates tag(s) + GitHub Release(s)
   with notes.
4. The fibertools `v0.X.Y` tag triggers cargo-dist → builds binaries + installer
   → uploads them to the release-please-created Release (because
   `create-release = false`).
5. `release: published` triggers `publish-crates.yml` → publishes MA (if
   changed) then fibertools to crates.io (idempotent; unchanged crates skipped).

### 4. cargo-dist changes

- `dist-workspace.toml`: add `create-release = false` under `[dist]`, then
  regenerate CI with `dist generate` (updates `release.yml` to upload artifacts
  to an existing release instead of creating one).
- Mark `molecular-annotation` with `dist = false` (in its
  `[package.metadata.dist]`) so its tag/release is a clean no-op for cargo-dist
  — it is a library with no binaries.

### 5. crates.io publish

- `publish-crates.yml` is unchanged in logic (fires on release published;
  idempotent per-crate publish in dependency order).
- Add a `concurrency` group so the two Releases created from one release-PR
  merge (fibertools + MA) don't race.

### 6. Retire git-cliff + update docs

- Remove `cliff.toml`; release-please owns changelog generation now.
- Rewrite `CONTRIBUTING.md` "Cutting a release": describe the release-please
  flow (merge conventional-commit PRs → review + merge the auto-opened release
  PR → automation tags, releases, builds binaries, publishes to crates.io).
  Remove the manual `cargo release` / `git tag` instructions and git-cliff
  preview steps. Keep an adapted "re-triggering a failed release" note.
- `release.toml` (cargo-release config, currently all-commented) can be removed
  since cargo-release is no longer the release driver.

### 7. What changes for contributors

- Stop hand-bumping versions and creating tags.
- Just merge Conventional-Commit PRs (squash recommended).
- Releases happen by merging the release PR release-please keeps open.
- On the rare PR that changes MA's code, path routing bumps MA automatically;
  no manual version edit needed.

## Risks / verification items

Prove these out during implementation, ideally on a throwaway pre-release tag
or a dry run, before relying on the flow:

1. **`exclude-paths` correctness** — confirm MA-only changes do NOT bump
   fibertools, and root changes DO. This option has historically been finicky
   (googleapis/release-please#2301).
2. **`cargo-workspace` plugin** — confirm a MA bump updates fibertools's
   `[workspace.dependencies]` requirement (and bumps fibertools) as intended,
   preserving the exact `^0.0.x` pin.
3. **cargo-dist no-op on MA tag** — confirm `dist = false` makes the
   `molecular-annotation-*` tag produce no artifacts and no errored jobs.
4. **Tag format + trigger** — confirm release-please emits `v0.X.Y` for
   fibertools and that cargo-dist's trigger/parse + `publish-crates.yml` accept
   it unchanged.
5. **Double-trigger safety** — confirm cargo-dist + `publish-crates.yml`
   behave when both a fibertools and an MA Release publish from one merge
   (concurrency guard; idempotent publish).

## Alternatives considered

- **release-please tags only; cargo-dist creates the Release (today's
  behavior).** Rejected: two changelog generators (git-cliff *and*
  release-please) would fight over the release notes.
- **Single release-please component (fibertools only); MA versioned manually.**
  Would give a single `CHANGELOG.md` and simplest wiring, with MA bumped by hand
  on its rare changes. Rejected by user preference in favor of release-please's
  monorepo mode with automatic per-crate versioning (accepting two changelogs).
- **Keep `cargo release` + git-cliff; don't adopt release-please.** Rejected:
  loses the release-PR review flow and automated path-based bumping.

## Out of scope

- No external announcement integrations (Slack/Discord/etc.) — none exist today
  and none are being added.
- No change to the crates.io Trusted Publishing setup beyond what
  `publish-crates.yml` already does (first-ever `molecular-annotation` publish
  still requires the documented one-time token bootstrap, tracked separately).
